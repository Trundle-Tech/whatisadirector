# What is a Director

An Electron desktop application built on a modern frontend stack using React, TypeScript, Vite, Tailwind CSS, and `shadcn/ui`.


## Quick Start Guide

### Prerequisites
Ensure you have [Node.js](https://nodejs.org/) installed on your machine.

### Installation
Clone the repository, then install the required npm dependencies:
```bash
npm install
```

### Running in Development
Start the concurrent development environment (starts the Vite dev server and launches Electron once the server is ready):
```bash
npm start
```

### Building the Project
Compile the React code and build the Vite assets:
```bash
npm run build
```

## Future Integrations & Steps

### Firebase Integration
The database is provisioned and now wired into the renderer through the Firebase SDK:
1. `src/lib/firebase.ts` initializes Firebase, supports standard Email/Password sign-in/sign-out, and exports Firestore.
2. `src/lib/firestore-store.ts` maintains real-time listeners for `/ingestionPackages` and read-only listeners for `/sops`, `/mops`, and `/eops`.
3. Ingestion submission writes new documents to `/ingestionPackages`.
4. Review actions update only the ingestion package record. Renderer code does not write directly to operational collections.
5. Firebase Authentication secures the application using standard Email/Password authentication via the `LoginForm` shell.
6. Vite env vars (`VITE_FIREBASE_API_KEY`, `VITE_FIREBASE_PROJECT_ID`, etc.) and Electron main process env vars (`VERTEX_API_KEY`, etc.) can be configured via a `.env` file based on [.env.example](file:///Users/nicklynch/Desktop/whatisadirector/.env.example).

### Operational Document System
The schema blueprints in `ideas_docs/schema/` define the Firestore data model. When implementing document features:
1. All operational documents extend `base_document.schema.json` — this is the non-negotiable foundation.
2. Each document type maps to its dedicated Firestore collection: `/sops`, `/mops`, `/eops`.
3. Every saved change must create a write-once record in `/{collection}/{docId}/versions/` — see `version_history.schema.json`.
4. No document may transition to `Active` status without a completed `approvalChain` entry in its version record.
5. See [schema_architecture.md](file:///Users/nicklynch/Desktop/whatisadirector/ideas_docs/schema/schema_architecture.md) before building any document-related feature.

### Google Vertex AI & Vector Graph
The backend includes full support for Google Vertex AI vector embeddings (`text-embedding-004`) and Gemini Chat (`gemini-2.5-flash`):
1. **Interactive Assistant**: Exposes IPC channels under `window.vertex.chat` for analyzing operational documents, performing safety checks, and suggesting gap fixes.
2. **Semantic Search & Vector Graph**: Exposes IPC channels under `window.vertex` for generating 768-dim embeddings, computing Cosine Similarity, and building a taxonomic/relational/semantic document graph.
3. **Local Testing**: Verify these integrations using the testing scripts in the scratch directory (e.g. `node scratch/test_vertex_embeddings.js`).

### Ingestion Tool
The ingestion tool is the primary path for creating and updating all operational documents. When implementing ingestion features:
1. The renderer process submits to `/ingestionPackages` — it never writes directly to operational collections.
2. All writes to `/sops`, `/mops`, `/eops` and their `versions` subcollections must go through the server-side Application Step (Cloud Function).
3. One approval covers everything in a package — primary document + all impacted document changes — atomically.
4. The Application Step is a Firestore transaction. If any part fails, the whole transaction rolls back.
5. See [ingestion_pipeline.md](file:///Users/nicklynch/Desktop/whatisadirector/ideas_docs/ingestion_pipeline.md) for agent behavior and [integration_guide.md](file:///Users/nicklynch/Desktop/whatisadirector/ideas_docs/integration_guide.md) for UI states and wiring before building any ingestion feature.
